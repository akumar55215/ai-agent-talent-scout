# TalentScout AI — Autonomous Talent Scouting & Engagement Agent

> Paste a Job Description → get a ranked shortlist of candidates, each with an explainable Match Score, a simulated outreach conversation, and an Interest Score interpreted from how they replied.

**Live demo:** _set in submission form_ · **Stack:** React 18 + Vite + TypeScript + Tailwind · Lovable Cloud (Supabase) edge functions · Lovable AI Gateway (Google Gemini 2.5 Flash) · GitHub public Search API.

---

## Why this exists

Recruiters spend the bulk of their time on two things:

1. **Sifting** through dozens of profiles to assess fit.
2. **Chasing** candidates to find out if they're even open to a conversation.

Both are bottlenecks with measurable cost. TalentScout AI compresses both into a single agent run: it parses the JD, discovers candidates, scores fit (with reasoning), simulates outreach, and produces a ranked shortlist where _Interest_ is a first-class signal — not a guess.

---

## What it does, end-to-end

1. **Parse JD** → structured fields (role, seniority, must/nice-have skills, YOE, location, domain) via tool-calling.
2. **Discover candidates** → curated pool of 15 realistic synthetic candidates **+** live GitHub user search filtered by required skills/location.
3. **Match & explain** → deterministic skill-overlap scoring + AI-generated rationale (strengths, gaps, one-line summary).
4. **Engage (simulated)** → for the top 5, the agent generates a personalized outreach message; the same model role-plays the candidate's reply using their persona; this happens for 4 turns.
5. **Score interest** → from the candidate's actual replies (not vibes), produce a 0-100 Interest Score + verdict (Hot / Warm / Cold / Not interested) + cited signals.
6. **Rank** → `combined = 0.6 × Match + 0.4 × Interest`. Recruiter sees who is _both_ qualified _and_ likely to engage.

### Why edge functions?
- The `LOVABLE_API_KEY` must never be exposed to the browser.
- All AI calls and any rate-limit/error handling live server-side.
- Functions are stateless — no DB persistence is required for the demo (intentional, per "real prototype" criterion).

### Why tool-calling for every AI call?
JSON-mode prompting is brittle. Tool-calling with a strict schema is the most reliable way to get back parseable, typed outputs from an LLM — critical for an agent pipeline where one bad parse breaks everything downstream.

## Scoring logic (explainable)

### Match Score (0-100)

Deterministic, fully explainable to a recruiter:

```
score = 0.55 × must_have_overlap_ratio
      + 0.15 × nice_to_have_overlap_ratio
      + 0.20 × experience_score          // 1.0 if ≥ required, decays linearly otherwise
      + 0.10 × title_alignment_score     // token overlap on titles
```

The skill set is normalized (lowercased, punctuation stripped) before comparison so `TypeScript` and `typescript` collapse. The AI layer adds a one-line summary, 2-4 strength bullets, and 0-3 gap bullets — but the **number is purely deterministic**, so a recruiter can audit it.

### Interest Score (0-100)

Generated by the LLM after a 4-turn simulated conversation, conditioned on:
- **Persona axes** (`openness`, `compFocus`, `remoteOnly`, `interests`) attached to each candidate.
- The actual JD details (location, domain, must-haves).

The model is instructed to score honestly — declines should produce low scores — and to cite 2-4 short signals quoting the candidate's own words. Anchored verdict bands keep scoring consistent: Hot 80+, Warm 55-79, Cold 30-54, Not interested <30.

### Combined Score
`0.6 × Match + 0.4 × Interest` — match is weighted higher because an interested-but-unqualified candidate is still a no-hire, but the weighting is a single line of code to tune.

## Candidate discovery strategy

| Source | How | Why |
|---|---|---|
| **Built-in pool** | 15 hand-crafted, diverse profiles (skills, geo, seniority) with personas | Guarantees a working demo with zero external dependencies; lets us test edge cases (declines, comp-focused, remote-only) |
| **GitHub Search API** | `language:<skill> location:<city> followers:>50` → fetch full user profile | Free, no auth required (60 req/h unauth — fine for a demo), real public data, real bios |

**Future sources** (intentionally out of scope for a 1-day prototype): LinkedIn (requires partnership), Wellfound/Otta (scraping), internal ATS imports (CSV upload). The discovery layer is a single edge function returning a normalized `Candidate` shape — adding sources is additive.

**Filtering**: per-source filtering happens at the API boundary (GitHub query string); the unified scoring step then ranks across sources uniformly so a strong GitHub hit can outrank a weaker pool candidate.

## Trade-offs we made

| Decision | Why | Cost |
|---|---|---|
| **Simulated candidates reply (vs. real outreach)** | Demo must run end-to-end with no inbox setup | Interest signal is plausible, not measured. Persona-grounding makes it directionally honest. |
| **Synthetic candidate pool + GitHub** | Free tier, runs anywhere, no scraping ToS issues | Doesn't reflect the volume of a real ATS — but the matching/scoring code is identical regardless of source. |
| **No persistence layer** | Faster iteration; the prototype is judged on the agent loop, not a CRUD app | A real product would store runs, conversations, recruiter feedback to fine-tune scoring. |
| **Single AI provider (Gemini 2.5 Flash via Lovable AI)** | One API key, free trial credits, fast | Locked-in for the demo. Provider abstraction is one fetch call away. |
| **Top-5 engagement only** | Cost & latency control (4 LLM calls per engagement × 5 = 20 calls) | Bottom of the list isn't engaged — but they're also the lowest-match candidates a recruiter would skip anyway. |
| **Deterministic match math + AI rationale** | Recruiter trust: the number can't hallucinate | Slightly less "smart" than a fully-LLM judge, but auditable. |

## Run it locally

```bash
# Prereqs: Node 18+, bun (or pnpm/npm)
git clone <repo-url>
cd talentscout-ai
bun install
bun run dev
```

Open `http://localhost:8080`. The hosted Lovable Cloud project is wired in via `.env` (`VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`) — the edge functions are already deployed. To run your own backend, fork the project in Lovable (it provisions Cloud automatically) or wire up Supabase manually and deploy the functions in `supabase/functions/`.

### Required server env vars (auto-provisioned in Lovable Cloud)
- `LOVABLE_API_KEY` — for the AI gateway
